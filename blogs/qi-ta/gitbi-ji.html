    # git 实战

    1. 创建文件夹 mkdir fileName
    2. 进入文件夹 cd fileName
    3. 初始化 git init
    4. 登录 git config --global user.name "xxx" git config --global user.email "xxxx@qq.com"
    5. 链接远程仓库 git remote add origin 远程仓库地址 git remote add origin https://gitee.com/Sinlon/front-end-of-prawn 、
    6. 拉取远程分支到本地：git fetch origin dev(dev是你要拉取的远程分支名称)
    7. 在本地创建dev分支并且切换到该分支：git checkout -b dev(本地分支名称)
    8. 拉取分支到本地：git pull origin dev(远程分支名称)
    9. 安装配置 cnpm install
    10. 创建主分支并切换到主分支 git checkout -b master
    11. 拉取远程主分支到本地 git fetch origin master
    12. 拉取主分支到本地 git pull origin master
    13. 在主分支上提交 git pull
    14. 切换到自身分支(dev为自身分支) `git checkout dev`
    15. 合并主分支到自身分支上 git merge master merge后为想要合并的分支名
    16. 填充项目
    17. 一次性把修改的代码放到版本库里 git commit --all -m "一些说明"
    18. 设置本地分支追踪远程分支 git push --set-upstream origin dev
    19. 提交入远程仓库 git push

    # 什么是HEAD

    在执行git init后，会在项目根目录下创建一个.git文件，此文件存储着对这个项目进行版本控制的所有信息

    ./git/refs/heads/ 文件夹存储着当前项目的所有分支的头指针，每一个头指针都指向该分支的最新提交。

    HEAD指针 ——–> 分支指针 ——–> 最新提交

    HEAD 的本质不是提向分支,而是指向 commit 提交。

    HEAD也有可能不用通过分支中转而是直接指向最新提交，但会进入一种特殊的状态 detached HEAD。当 HEAD 指针直接指向提交时，就会导致 detached HEAD 状态。在这个状态下，如果创建了新提交，新提交不属于任何分支。相对应的，现存的所有分支也不会受 detached HEAD 状态提交的影响。

    git checkout <commit id> //将HEAD切换到指定的提交ID HASH
git checkout --detach    //进入detached HEAD 状态。

    # 需求方法

    # 在工作区修改后反悔如何将暂存区的内容覆盖还原work dir的内容

    git checkout .  // .代表通配符，将所有暂存区内容覆盖work dir 区域内容
git checkout a.txt  //指定文件还原
    1
    2

    注意

    • 如果在工作目录新建了文件，是无法通过此命令而删除新增加的文件
    • 此命令无法还原，在执行前确保work dir 的修改可以抛弃

    # 在提交文件时，有时候我们刚刚commit提交完，这时发现漏掉了几个文件没有添加，或者可能提交信息写错了，想要修改提交信息或者新增提交文件但又不想因此新建个commit

    git add .   
git commit --amend   //会将缓存区中的文件提交。做完上次提交后，如果发有未提交的文件，可以将其添加到缓存区，再执行此命令修改提交信息即可。
    1
    2

    # 将history区的文件还原到stage区

    即当进行了 git add .命令后发现还有文件未完善，想还原stage内容，可以通过将history的内容覆盖到stage上实现还原

    git reset a.txt    //指定还原文件  简写
git reset --mixed HEAD a.txt
    1
    2

    该命令的自然语言描述是：不改变work dir中的任何数据，将stage区域中的a.txt文件还原成HEAD指向的commit history中的样子。就相当于把对a.txt的修改从stage区撤销，但依然保存在work dir中，变为unstage的状态。

    进行此操作前确定stage区域的文件可以抛弃

    # 将history的内容还原到工作目录中

    比如从 GitHub 上clone了一个项目，然后乱改了一通代码，结果发现我写的代码根本跑不通，于是后悔了，干脆不改了，我想恢复成最初的模样，怎么办？

    git checkout HEAD .
    1

    这样，work dir和stage中所有的「修改」都会被撤销，恢复成HEAD指向的那个history commit。

    注意，类似之前通过stage恢复work dir的checkout命令，这里撤销的也只是修改，新增的文件不会被撤销。

    当然，只要找到任意一个commit的 HASH 值，checkout命令可就以将文件恢复成任一个history commit中的样子：

    $ git checkout 2bdf04a some_test.go
Updated 1 path from 2bdf04a
git log  //可以查看提交历史
    1
    2
    3

    比如，我改了某个测试文件，结果发现测试跑不过了，所以就把该文件恢复到了它能跑过的那个历史版本……

    此操作会覆盖work dir 内容，无法还原，执行前确认work dir 的文件可以抛弃

    # 其他技巧

    # 合并多个commit

    比如说我本地从17bd20c到HEAD有多个commit，但我希望把他们合并成一个commit推到远程仓库，这时候就可以使用reset命令：

    $ git reset 17bd20c
$ git add .
$ git commit -m 'balabala'
    1
    2
    3

    回顾一下刚才说的reset命令的作用，相当于把 HEAD 移到了17bd20c这个commit，而且不会修改work dir中的数据，所以只要add再commit，就相当于把中间的多个commit合并到一个了。 即从 17bd20c 提交 到 最新的分支提交这部分的提交合并

    # 需求二，由于HEAD指针的回退，导致有的commit在git log命令中无法看到，怎么得到它们的 Hash 值呢？

    再重复一遍，只要你不乱动本地的.git文件夹，任何修改只要提交到commit history中，都永远不会丢失，看不到某些commit只是因为它们不是我们当前HEAD位置的「历史」提交，我们可以使用如下命令查看操作记录：

    $ git reflog  //查看所有提交信息hash
    1

    比如reset，checkout等等关键操作都会在这里留下记录，所有commit的 Hash 值都能在这里找到，所以如果你发现有哪个commit突然找不到了，一定都可以在这里找到。

    # Git问题及其解决方法

    # 本地创建项目再在码云创建仓库导致的无法合并推送的问题

    正常应该先创建仓库，初始化连接后再在相应文件创建项目

    # 问题原因&解决步骤

    1. 因为本地的仓库和远程的实际上是两个不同的仓库，执行如下命令忽略合并

      git pull origin master --allow-unrelated-histories (该选项可以合并两个独立启动仓库的历史)
      1

    2. 如果通过克隆的方式建立起本地仓库则不会有此问题

    # git rebase

    git pull将以merge远程分支拉到本地，如果使用 git pull -r，会以 rebase方式拉取远程修改到本地，merge方式合并的分支会有很多「分叉」，而rebase方式合并的分支就是一条直线。

    # git rebase实现方式

    • git rebase master 在哪个分支执行此命令，就会将该分支接到master分支上。
    • 而master分支又会接到该分支和master分支的最近公共祖先（LCA）上，如果有冲突，会提示解决冲突。

    # 假设Git目前只有一个分支master。开发人员的工作流程是

    • git clone master branch 从项目地址克隆连接仓库
    • 在自己本地checkout -b local创建一个本地开发分支
    • 在本地的开发分支上开发和测试
    • 阶段性开发完成后（包含功能代码和单元测试），可以准备提交代码
      • 首先切换到master分支，git pull拉取最新的分支状态 然后切回local分支
      • 通过git rebase -i 将本地的多次提交合并为一个，以简化提交历史。本地有多个提交时,如果不进行这一步,在git rebase master时会多次解决冲突(最坏情况下,每一个提交都会相应解决一个冲突) 合并本地提交
      • git rebase master 将master最新的分支同步到本地，这个过程可能需要手动解决冲突(如果进行了上一步的话,只用解决一次冲突) 将主分支和本地分支合并
      • 然后切换到master分支，git merge将本地的local分支内容合并到master分支
      • git push将master分支的提交上传
      git rebase -i  [startpoint]  [endpoint]
    1

    其中-i的意思是--interactive，即弹出交互式的界面让用户编辑完成合并操作，[startpoint] [endpoint]则指定了一个编辑区间，如果不指定[endpoint]，则该区间的终点默认是当前分支HEAD所指向的commit(注：该区间指定的是一个前开后闭的区间)。 在查看到了log日志后，我们运行以下命令：

    git rebase -i HEAD~3
    1

    不要通过rebase对任何已经提交到公共仓库中的commit进行修改
